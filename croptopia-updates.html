Building Croptopia, Part 1: Bolting a Game onto a Solana Wallet

Croptopia · Devlog · 9 min read

Six months ago I decided to build a farming game where the land was real. By "real" I meant: you'd own it on Solana, you'd lock value into it on claim, and no server admin could ever evict you. That single decision is where everything else followed from.

This is the first of three devlog posts about how I built Croptopia. I'm writing them now, after mainnet, because I want to remember what was hard before I forget. The launch post comes last.

The framework problem

Most "web3 game" projects I've seen either ship a Unity WebGL build with a 20MB bundle that takes thirty seconds to load on cold cache, or they write everything in raw Canvas and end up with a glorified spreadsheet. Neither of those felt right for a top-down farming game that needed to feel snappy and look like an SNES title.

I picked Phaser 3. It's JavaScript-native, designed for 2D games, with a tile-based scene system that fits a 10×10 plot grid perfectly. The trade-off is that Phaser has zero opinions about anything else in your app — it wants the whole page.

I needed React for everything around the canvas — modals, wallet connection, leaderboards, tribes, market UI. Without React, all of that becomes manual DOM work. Without Phaser, the game world is a static grid. So the question became: can I get the two to coexist?

First attempt: props and callbacks

The naive approach is to mount Phaser inside a React component and pass callbacks down as props. Click a plot, the Phaser scene calls onPlotClick(plotId), React opens a modal.

function GameWorld({ onPlotClick, wallet }) {
  const containerRef = useRef(null);

  useEffect(() => {
    const game = new Phaser.Game({
      parent: containerRef.current,
      scene: new WorldScene({ onPlotClick, wallet }),
    });
    return () => game.destroy(true);
  }, []);

  return <div ref={containerRef} />;
}

This works for about thirty seconds. Then you realize the scene was constructed with the first value of onPlotClick and wallet. When React re-renders with new values, the scene doesn't know. Click a plot and the modal opens with the wrong wallet.

You can fix this by adding the props to the useEffect dependency array. But then Phaser tears down and rebuilds the entire game every time those values change. The world flickers. Sprites reload. It's unusable.

Second attempt: event emitter

I tried a pub/sub bus next. React subscribes, Phaser emits. The scene doesn't hold references to anything mutable — it fires bus.emit('plot:click', plotId) and trusts that whoever is listening will handle it.

This worked, technically. But debugging it was painful. Every event went through one channel. When something didn't fire, I had no idea whether the emitter was wrong, the subscriber was unmounted, or the event name was misspelled. I was adding console.log to a global event bus and grepping output.

I also kept tripping over React 18 strict mode's double-mount, where my subscriptions registered twice and modals opened twice. I "fixed" it with cleanup logic that worked in dev but broke in prod, where strict mode wasn't active. Two hours I'm not getting back.

Third attempt: a global object on globalThis

The solution I ended up with is the dumbest thing I could have done, and it's been bulletproof for months.

I put a single object on globalThis called __cw (Croptopia Wrapper). Phaser registers default no-op methods when the scene is created. React, when it mounts, replaces those methods with closures over current state. When state changes, React replaces the methods again. Phaser doesn't care, because the property lookup happens at call time.

// In WorldScene.create()
globalThis.__cw = {
  wallet: null,
  onClick: (plotId) => {},
  focusPlot: (plotId) => {},
  refreshAnimals: (plotId, type) => {},
  refreshFarmers: (plotId, count) => {},
  // ...
};

// In React
useEffect(() => {
  if (!globalThis.__cw) return;
  globalThis.__cw.wallet = wallet;
  globalThis.__cw.onClick = (plotId) => {
    setSelectedPlot(plotId);
    setModalOpen(true);
  };
}, [wallet]);

That's it. No event bus, no provider, no subscription model. When the Phaser scene wants to tell React something happened, it calls globalThis.__cw.onClick(plotId). When React wants to tell Phaser to spawn a cow, it calls globalThis.__cw.refreshAnimals(plotId, 'cow'). Both directions go through the same flat object.

The reason it works: property lookup is lazy. Phaser doesn't capture a reference to the function — it captures a reference to the object and looks up the function each call. When React replaces the function, Phaser sees the new one immediately. And there's no subscription model, so nothing to register, clean up, or double-fire.

If your bridge between two systems is simple enough to fit on a sticky note, you can debug it. If it isn't, you can't.

The accidental affordance

Because the bridge is just a global, I can poke at it from the browser console. When I'm debugging a sprite issue, I open devtools and type __cw.refreshAnimals(42, 'cow') and watch a cow appear. When I'm testing the plot focus animation, I type __cw.focusPlot(7) and the camera pans.

I don't need a debug menu in the game UI. I don't need a hidden hotkey. The bridge is the debug menu. This is the kind of unplanned benefit you only get from doing the dumb thing first.

Building Croptopia, Part 2: The Day Plots Became Stakes

Croptopia · Devlog · 10 min read

The first playable version of Croptopia let anyone claim a plot for free. You'd hit a button, sign one transaction, and the plot was yours forever. That version played terribly. This is the post about why, and how I fixed it.

The free-claim trap

The original design was inspired by what I'd seen in other on-chain games: claim a free plot, then pay for upgrades, fertilizers, animals, whatever. The plot itself was the on-ramp. I figured the economy could happen in the layer above the land.

I ran a closed playtest with eight wallets. Within forty-five minutes, three of those wallets had claimed seventy of the hundred plots — and were sitting on them, ignoring the game. They weren't farming. They weren't selling. They were just holding empty land, waiting to see if it would be worth anything later. The other five players had nowhere to claim. The world looked dead.

This is the well-known free-claim trap. If a thing costs nothing to acquire, the optimal strategy is to grab as much of it as you can on day one and figure out the rest later. I'd built a game that rewarded squatting.

The pivot

The fix was obvious in hindsight: make holding a plot cost something. Make leaving a plot cost more.

I introduced a USDC stake on claim. To grab a Bronze plot, you lock 1 USDC in the protocol. Silver costs 5. Gold costs 15. Diamond costs 50. The stake stays locked for as long as you hold the plot — it doesn't get burned, it isn't a fee, it's collateral. If you transfer the plot to someone else, your stake unlocks; the new owner posts a new one.

The breaking change came with the abandon mechanic. If you walk away — sign the abandon_plot transaction yourself — your stake doesn't return to you. Half burns. Half goes to the protocol treasury.

pub fn abandon_plot(ctx: Context<AbandonPlot>) -> Result<()> {
    let plot = &mut ctx.accounts.plot;
    require!(plot.owner == ctx.accounts.player.key(), ErrorCode::NotOwner);

    let stake    = plot.locked_amount;
    let burn     = stake / 2;
    let treasury = stake - burn;

    transfer_from_vault(&ctx, ctx.accounts.burn_vault,     burn)?;
    transfer_from_vault(&ctx, ctx.accounts.treasury_vault, treasury)?;

    plot.owner         = Pubkey::default();
    plot.locked_amount = 0;
    Ok(())
}

That single function — abandon_plot — is the line that made Croptopia feel like a game with real stakes instead of a clicker with NFTs.

Calibrating the ladder

1 / 5 / 15 / 50 USDC didn't come from nowhere. I needed the bottom rung to be low enough that someone could try the game on a whim, and the top rung to be high enough that whales actually had to commit. The exponential ladder (roughly 5×, 3×, 3.3×) gives each tier a noticeably different commitment level.

I also added a 5-plot cap per wallet, enforced in the program:

require!(player.plot_count < MAX_PLOTS_PER_WALLET, ErrorCode::TooManyPlots);

Five plots is enough to build a serious farm. It isn't enough to corner the market. Without this cap, a single 250-USDC wallet could buy every Diamond plot and force everyone else into Bronze permanently.

The second playtest

Same eight wallets, same hour. Different game.

The hoarders from the first playtest didn't show up. The two who did try to grab multiple Diamond plots stopped at one or two — 50 USDC felt different from "click to claim." The Diamond plots filled within forty minutes, but spread across five different wallets. Silver was the middle-ground sweet spot. Bronze sold out quickly because new players used it as an on-ramp.

Within an hour, the world had:

  • Eight active farmers — every player engaged
  • Five Diamond plots, each held by a different wallet
  • Three abandons (people who claimed Bronze, didn't like it, walked away — collectively burning 1.5 USDC and feeding 1.5 to treasury)
  • Active trading between players because plots felt scarce

The same group of testers behaved completely differently because the asset finally had weight.

Why this works

A stake does what free claims can't: it filters. Anyone willing to lock 50 USDC into a Diamond plot has decided they actually believe in the game. The market for plots becomes a market for conviction, not a queue of wallet-spam.

The abandon-burn isn't punishment. It's the cost of being wrong about your conviction. If you commit 50 USDC to a Diamond plot and decide the game isn't for you, you lose 50 USDC. That's a real signal. The system reads it as this slot just freed up for someone who'll use it.

The game's design didn't change. The cost of holding changed. The behavior followed.

Building Croptopia, Part 3: An Order Book in 100 Lines of Rust

Croptopia · Devlog · 9 min read

Once players had plots and harvests, they wanted to trade. The first version of Croptopia had a fixed-price NPC vendor — sell wheat for 2 USDC, magic herbs for 50, take it or leave it. It worked. It was also boring.

This is the post about ripping that out and replacing it with a real on-chain order book. And the bug I shipped, accidentally, on day one.

Why fixed prices were wrong

Fixed-price vendors are tempting because they're simple. You write a constant, you check inventory, you transfer USDC, you're done. No matching engine, no escrow, no state.

The problem is that fixed prices flatten the world. Wheat doesn't get more valuable when there's a Golden Hour rally. Magic herbs don't spike when a tribe goes raiding. The economy can't react to itself. Every commodity prices at exactly what the NPC says it prices at, forever.

What I wanted was: let players set the price. Place a bid for 50 wheat at 1.8 USDC each. Place an ask for 30 wheat at 2.1. When the books cross, the protocol settles the trade. The market discovers its own clearing price every minute.

The Order account

The shape ended up small. Each order is one Anchor account:

#[account]
pub struct Order {
    pub maker:      Pubkey,
    pub item:       ItemType,        // Wheat | Corn | Egg | Milk | ...
    pub side:       Side,            // Bid | Ask
    pub quantity:   u64,
    pub remaining:  u64,
    pub price:      u64,             // USDC per unit
    pub status:     OrderStatus,
    pub created_at: i64,
}

#[derive(AnchorSerialize, AnchorDeserialize, Clone, Copy)]
pub enum Side         { Bid, Ask }
#[derive(AnchorSerialize, AnchorDeserialize, Clone, Copy)]
pub enum OrderStatus  { Open, Filled, Cancelled }

When you place a sell order (an ask), the program locks your wheat in an item escrow account. When you place a buy order (a bid), the program locks your USDC. The escrows are PDAs; only the program can move tokens out, and only when the matching engine fires.

The matching engine

The match logic is straightforward in shape, ruthless in execution. Walk asks lowest-first. Walk bids highest-first. While the best bid is at or above the best ask, fill at the ask price. Stop when the books cross or one side runs out.

pub fn match_book(ctx: &Context<PlaceOrder>, item: ItemType) -> Result<()> {
    let mut asks = best_asks(item, 10)?;
    let mut bids = best_bids(item, 10)?;

    for ask in asks.iter_mut() {
        for bid in bids.iter_mut() {
            if bid.price < ask.price        { break; }
            if bid.maker == ask.maker        { continue; }   // self-match guard

            let qty   = ask.remaining.min(bid.remaining);
            let price = ask.price;
            let gross = qty * price;
            let fee   = gross * 25 / 1000;       // 2.5%
            let net   = gross - fee;

            settle_payment(ctx, ask.maker, net, fee)?;
            settle_delivery(ctx, bid.maker, item, qty)?;

            ask.remaining -= qty;
            bid.remaining -= qty;
            if ask.remaining == 0 { ask.status = OrderStatus::Filled; }
            if bid.remaining == 0 { bid.status = OrderStatus::Filled; }
            if ask.remaining == 0 { break; }
        }
    }
    Ok(())
}

About a hundred lines once you count the helpers. The 2.5% fee splits between treasury and burn vault — same pattern as the abandon mechanic. The patient seller wins ties (we fill at the ask price, not the midpoint, which means a buyer who shows up later doesn't move the price up just by being there).

The bug I shipped

You'll notice this line:

if bid.maker == ask.maker { continue; }   // self-match guard

That comment was not in the original commit. I added it after a wallet fake-pumped the daily volume by 4,000 USDC in twenty minutes by placing matching bids and asks against itself.

Here's what happened: a player figured out that without a self-match guard, they could place a 100-wheat ask at 2.0 USDC and a 100-wheat bid at 2.0 USDC from the same wallet. The matching engine would happily fill them against each other. They'd pay the 2.5% fee, but the volume would show up on the public board.

The "trade" was: ask escrow holds 100 wheat → bid escrow holds 200 USDC → match fires → wallet gets 200 USDC × 0.975 = 195 USDC back, and 100 wheat back. Net cost: 5 USDC. Net benefit: 200 USDC of fake volume on the public chart.

I noticed it on day two when the volume looked suspicious. The fix was one line.

if bid.maker == ask.maker { continue; }

Self-match guards are basic order book hygiene. I shipped without one because I'd never built an order book before. Live and learn. The wallet that exploited it didn't gain anything material (the volume metric doesn't gate anything) but it would have, if anyone had built a leaderboard ranked by volume. The patch went out three hours after I noticed.

What the order book does for the world

Wheat doesn't have a fixed price anymore. It has the price the highest current bid is willing to pay. Magic herbs spike when supply dries up and crater when a Diamond plot dumps inventory. Players watch the book. Tribe leaders coordinate buys. Alliances trade against each other.

The market makes itself. I wrote the rules. The players write the prices.

The hardest part of designing an economy is getting out of the way once you've set the rules.

Croptopia Is Live on Solana

Croptopia · Launch · 6 min read

Six months. Three pivots. One self-match bug. Croptopia is live on Solana.

The protocol is deployed, the world is open, and the first farmers are already farming. If you've been following the devlog posts about the React-Phaser bridge, the stake mechanic, and the order book — this is the post where it all goes live.

What it is, one more time

100 farming plots on Solana, each one a USDC-staked position you actually own. Plant crops, raise animals, defend your land, trade everything on an on-chain order book. The world is finite, the stakes are real, the protocol code is auditable in the public repo.

The world

Fixed at 100 plots across four tiers. They don't mint, don't airdrop, don't expand. Once they're claimed, they're claimed.

TierSupplyStakeYieldCrop / Animal Slots
Bronze501 USDC1.0×4 / 1

Diamond550 USDC3.0×10 / 5

Each wallet is capped at five plots. The cap is enforced at the program level, not the UI.

The stake

USDC is the unit of account. When you claim a plot, your stake locks into a per-plot vault PDA controlled by the Croptopia program. The vault holds it for as long as you hold the plot. Walk away and half burns, half goes to the treasury. The plot returns to world supply.

Land you don't believe in, you don't claim. Land you do, you defend.

The token

$CROP is the community token, live on Pump.fun. It isn't the unit of account for the game (USDC is) — it's a coordination token, used for snapshot eligibility on upcoming drops. The biggest one is the free Farmer NFT mint coming next month: 1,000 unique on-chain workers, free for $CROP holders, full details ahead of the snapshot block.

How to start

  1. Connect a Solana wallet — Phantom, Backpack, and Solflare all work.
  2. Open croptopia.world, pick a plot, claim it. One transaction.
  3. Plant something. Wheat on every Bronze plot is a fine starting move. Harvest in five minutes. Sell on the order book. Iterate.

The roadmap

  • This month: Farmer NFTs (free mint for $CROP holders)
  • Next month: On-chain raid mechanic with defender stacking and asynchronous replays
  • Q4: Plot leasing (rent yield rights without selling) and cross-tribe coordination contracts
  • 2027: Mobile companion app, cosmetic plot upgrade NFTs

Each feature ships when the contracts are stable, not on a calendar.
